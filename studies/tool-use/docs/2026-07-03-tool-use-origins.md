# Tool Use 的身世:在「official 機制」出現前,好幾組人馬各自逼近同一個答案

`tool-use-basics.md` 講的是「tool use 是什麼、為什麼需要它」,概念上不牽涉時間。這篇補的是
**時間軸**:這個機制不是某一天被發明出來的,是好幾條各自獨立的研究線,在 2023 年中被廠商收編
成官方機制。查證過程中修正了兩次自己原本的猜測——過程本身也留在下面,不是最後才修過的乾淨版本。

## 學術先驅期(2021-2023):四條各自獨立的線

| 時間 | 是誰 | 做了什麼 | 信心 |
|---|---|---|---|
| 2021-12-17 | [WebGPT](https://arxiv.org/abs/2112.09332)(OpenAI) | 微調 GPT-3,用固定指令集(搜尋/點擊/引用)操作一個文字版瀏覽器環境,靠模仿學習 + RLHF 訓練——查得到、有論文的最早案例 | VERIFIED |
| 2022-04-04 | [SayCan](https://arxiv.org/abs/2204.01691)(Google) | LLM 對一份固定的機器人技能清單(~551 個)打分數,跟一個學到的可行性函數相乘去挑動作——是「選」不是「生成呼叫」 | VERIFIED |
| 2022-05-01 | [MRKL Systems](https://arxiv.org/abs/2205.00445)(AI21) | LLM 當路由器,把問題分派給專家模組,而且真的做出來了(Jurassic-X) | VERIFIED |
| 2022-10 | ReAct(arXiv:2210.03629,Princeton/Google) | 第一個**免微調、套在任意工具上都行**的技巧:交替生成「Thought」跟「Action」,自由文字,外部程式碼自己解析——因為通用,後來被引用最多 | VERIFIED |
| 2023-02-09 | [Toolformer](https://arxiv.org/abs/2302.04761) | 換個方向:直接訓練模型自己學會吐工具呼叫格式,不靠 prompting 技巧 | VERIFIED |
| 2023-05-24 | [Gorilla](https://arxiv.org/abs/2305.15334) | 微調開源模型取得呼叫能力 | VERIFIED |
| 2023-07-31 | [ToolLLM](https://arxiv.org/abs/2307.16789) | 同上路線,更大規模的工具集 | VERIFIED |

**順序確認**:WebGPT(2021-12) < SayCan(2022-04) < MRKL(2022-05) < ReAct(2022-10)。ReAct
不是最早的,是**第一個通用化、不用微調就能套用的技巧**,這是它後來被當成「起點」講的真正原因,
不是因為它真的最早。

另外查了一個「Gwern 用 GPT-3 操作瀏覽器買 AirPods」的說法,**沒有查到可信來源**,無法確認是否
真的存在或年份,標記為未證實,沒有放進正式時間軸。

**這整段(2021-2023 上半)就是「土法煉鋼時代」**:每個團隊各自發明一套自己的暗號——ReAct
團隊約定「Thought: ... Action: ...」這種格式,WebGPT 團隊乾脆把模型整個微調成只認識瀏覽器
指令,MRKL 團隊設計了一套路由架構。這些暗號系統各自獨立、互不相容,沒有一套是官方的、通用
的——開發者想做「LLM 呼叫工具」,得自己選一套暗號、自己寫解析程式碼去猜模型吐出的文字裡藏
的是不是一個動作。**結束這個時代的不是某篇論文,是下面這件事。**

## 廠商官方機制搶跑(2023):是 OpenAI 讓 tool use 收斂成下一個時代

**土法煉鋼時代的下一個時代,是「官方統一機制」的時代,而讓它發生的是 OpenAI。**
2023-06-13 之前,「LLM 呼叫工具」這件事沒有共同標準,是一堆互不相容的暗號系統;那一天之後,
開發者不用再自己發明暗號、自己寫正規表達式解析,直接用 API 內建的 `tools` 參數,模型原生吐出
結構化的呼叫物件——**收斂的動作是 OpenAI 做的,時間點就是那一天**。Anthropic
(2023-11-21 beta)、Google(2023-12-13)是在 OpenAI 開了第一槍之後才跟進推出自己的官方
版本,都晚了五到六個月。

| 時間 | 廠商 | 事件 | 信心 |
|---|---|---|---|
| **2023-06-13** | **OpenAI** | Function Calling 正式上線——業界第一個官方機制 | VERIFIED(獨立第三方同日報導交叉確認,官方原文連結被 403 擋下無法直接抓取) |
| 2023-11-21 | Anthropic | Tool use(beta),隨 Claude 2.1 一起發布,比 OpenAI 晚約 5 個月 | VERIFIED(直接讀到 anthropic.com/news/claude-2-1 原文) |
| 2023-12-13 | Google | Gemini API 上線即含 function calling,比 OpenAI 晚約 6 個月 | VERIFIED(直接讀到 blog.google 原文) |
| 2024-05-30 | Anthropic | Tool use 從 beta 轉正式 GA(Claude 3 全系列 + API/Bedrock/Vertex) | LIKELY(來源是搜尋摘要 + 同日官方推特,沒有直接抓到 blog 原文) |

三家的官方公告都**沒有互相點名**對方——沒有查到 Anthropic 或 Google 的文案提到「因為 OpenAI
先做了 X,所以我們也做」這種話,都是包裝成自家獨立能力發布的。

## 反轉:function calling 先上線,structured output 這個通用功能是 14 個月後才長出來的

這是這篇查證過程中修正過的地方,值得特別說清楚,因為原本的猜測**方向是反的**。

一開始以為「structured output 是更通用的底層技術,function calling 是拿它去做的一種特例」,
邏輯上聽起來合理,但**歷史上發生的順序不是這樣**:

| 時間 | 功能 | 保證等級 |
|---|---|---|
| 2023-06-13 | Function Calling | 模型自己決定要不要呼叫、填什麼參數,**沒有 schema 強制保證**——格式偶爾還是會錯 |
| 2023-11-06 | JSON mode | 保證輸出「語法上合法的 JSON」,但不保證符合你指定的 schema | LIKELY(社群/文檔交叉確認,沒抓到專屬公告原文)
| **2024-08-06** | **Structured Outputs** | 保證輸出**完全符合你指定的 JSON Schema**(官方 evals 測出 100%),同時套用在「工具呼叫的參數」跟「一般回答格式」兩種情境——這才是真正通用化的版本 | VERIFIED(標題與日期經獨立第三方同日文章交叉確認) |

也就是說:**先有一個專門為了「觸發工具呼叫」而生、格式沒有強保證的機制(2023-06),14 個月後
才從裡面長出一個「格式完全保證、任何情境都能用」的通用版本(2024-08)**。方向是「專用先出現,
通用後出現」,不是反過來。`tool-use-basics.md` 裡「function calling 是 structured output
的一種特定用法」這句話,講的是**概念上**兩者的關係(A 確實是 B 的一種特定用法),跟這裡的
**時間先後**是兩件事——概念上的包含關係不代表通用版本一定先誕生。

## Perplexity:不等官方機制,自己土法煉鋼先上線產品

[Perplexity](https://research.perplexity.ai/articles/architecting-and-evaluating-an-ai-first-search-api)
在 **2022-12-07** 上線第一版產品("Perplexity Ask"),比 OpenAI 官方 function calling
(2023-06)還早半年——用的是自己客製的 RAG pipeline(LLM + 自製搜尋/檢索),完全不依賴任何
廠商的官方 tool-use API。(LIKELY——多個第三方來源一致,沒有抓到 Perplexity 自己 2022 年當下
發布的原文,他們自己寫的架構文章是事後回顧,不是當時的公告。)

這是跟前面 WebGPT / MRKL 那類學術先驅同一種故事,只是這次是拿去做成真正上線的消費產品:**概念
比官方機制早,產品也比官方機制早**——不用等廠商把機制做成標準,自己就能土法煉鋼湊出一個堪用的
版本先上線。

## 現在(2026):原生訓練取代了 prompt 教格式,但這不是故事的起點——WebGPT 早就是訓練出來的

前面整段時間軸,講的都是「怎麼讓模型呼叫工具」這件事本身怎麼從研究走向官方機制。還有一層
更底層的變化沒講到:**模型怎麼「學會」呼叫工具格式,這件事本身走的是一條三段式弧線,不是
兩段式。**

容易漏掉的地方:**ReAct 不是「訓練 vs prompt」這條線的起點**——上面表格裡最早的 WebGPT
(2021-12)本身就是**訓練出來的**(微調 GPT-3 + RLHF),不是靠 prompt 教格式。差別在
「廣度」:WebGPT 訓練出來的能力**很窄**,只認瀏覽器那幾個固定指令,想換一種工具就得整個
重新訓練一次。ReAct(2022-10)走的是完全相反的路線:**不訓練,但廣**——免微調、套在任意
工具上都行,代價是每次都要在 prompt 裡塞範例教格式,而且格式不保證跟得住。

**現在的原生 function calling,是把這兩條路線的優點合在一起**:模型在訓練階段(SFT,越來越
多加上 RL)直接餵大量「工具呼叫軌跡」(使用者問題 → 中間推理 → 呼叫工具 → 工具結果 → 最終
答案),tokenizer 甚至有專門的特殊 token 標記工具呼叫——但學到的不是「認識瀏覽器」這種窄
技能,是「看到任何一份工具 schema,就知道怎麼填」這種通用技能。三段式弧線是:**窄且訓練
(WebGPT)→ 廣但要教(ReAct)→ 廣且訓練(現在)**。

這件事的種子其實在學術先驅期就已經種下——上面表格裡的 **Toolformer(2023-02)**,論文標題
直接就是「Language Models Can Teach Themselves to Use Tools」,核心概念就是這個。差別是:
2023 年那是一篇侷限在小規模、固定工具集的研究論文;**2026 年已經變成所有前沿模型的標準做法**,
規模擴大到任意工具、任意場景都適用。

**但有一個地方不能講得太滿:格式不用教了,不代表完全不用給任何輸入。** 「宣告有哪些工具可以用」
這件事還是要做——`tools` 參數(name/description/schema)還是得送進 API 請求裡,只是這不算
傳統意義的 prompt(不是自由文字、不是 few-shot 範例),是結構化的 API 參數。精確的說法是:
「教格式」這件事被訓練取代了,「告訴模型現在有什麼工具」這件事還是要做,只是換了一種形式。

**誠實聲明**:Anthropic、OpenAI、Google 都**沒有公開揭露**自己訓練 tool use 的確切配方(軌跡
資料量、SFT/RL 比例)——查到的是研究社群的方法論共識(SFT + RL、tokenizer 特殊 token 標記
工具呼叫),不是特定廠商的官方揭露。另外查到一篇研究(arXiv:2508.05118)指出純 SFT 容易讓
模型變成「表面模式匹配」而非真正穩健的推理能力,RL 是目前研究社群還在推進的方向,不是已經
解決的問題。

## 訓練出來的原生能力,好不好要有人量測——BFCL

上一節留了一個空隙:模型現在原生就會呼叫工具了,但**怎麼知道訓練出來的這個能力做得好不好**?
答案是 [Berkeley Function-Calling Leaderboard(BFCL)](https://gorilla.cs.berkeley.edu/leaderboard.html)
——**2024-02** 上線,做的人就是這篇時間軸裡已經出現過的 **Gorilla 團隊**(Berkeley)。

這是一條漂亮的閉環:Gorilla 團隊 2023-05 先解決「怎麼做出一個會呼叫工具的模型」,9 個月後
(2024-02)同一批人回頭解決「怎麼知道做得好不好」,做出用 AST(抽象語法樹)比對呼叫格式對不
對的評測基準,後來陸續加上多輪對話評測、真正執行呼叫的 executable track。這個基準後來成為
業界拿來比較各家模型「原生工具呼叫」能力的共同量尺——不是某一家廠商自己說了算。
